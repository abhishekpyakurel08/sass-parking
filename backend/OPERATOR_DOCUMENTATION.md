## 2. Authentication

All operator endpoints require a valid **Bearer token** in the `Authorization` header.

### Login

```
POST /api/v1/auth/login
```

**Request Body:**
```json
{
  "email": "operator@yourfacility.com",
  "password": "your_secure_password"
}
```

**Success Response `200`:**
```json
{
  "success": true,
  "message": "Login successful",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "expires_in": "15m",
    "user": {
      "id": "664a1b...",
      "name": "Raju Operator",
      "email": "operator@yourfacility.com",
      "role": "GATE_STAFF",
      "tenant_id": "664a1a...",
      "gate_assignment": "ENTRY",
      "ticket_prefix": "GTE1"
    }
  }
}
```

> **Important:** The `access_token` expires quickly (typically 15 minutes). Use the refresh endpoint to get a new one without re-logging in.

### Refresh Token

```
POST /api/v1/auth/refresh
```

The refresh token is automatically set as an `HttpOnly` cookie on login. The server will issue a new access token if the refresh token is valid. Token rotation is enforced — reused tokens trigger a security lockout.

**Response `200`:**
```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "expires_in": "15m"
  }
}
```

### Logout

```
POST /api/v1/auth/logout
Authorization: Bearer <access_token>
```

Invalidates the current session and clears the refresh token cookie.

## 3. Gate Assignment & Restrictions

Each `GATE_STAFF` account is assigned a `gate_assignment` field set by the Tenant Owner:

| Assignment | Can Check-In | Can Check-Out / Process Payment |
|------------|:------------:|:-------------------------------:|
| `ENTRY`    | ✅            | ❌                               |
| `EXIT`     | ❌            | ✅                               |
| `BOTH`     | ✅            | ✅                               |
| *(unset)*  | ✅            | ✅                               |

If an `ENTRY`-only operator attempts a check-out or payment, the server returns:
```json
{
  "success": false,
  "message": "Access denied: You are only authorized to process entries."
}
```

### Ticket Prefix
Each operator also has a `ticket_prefix` (e.g., `"GTE1"`). All tickets generated by this operator are prefixed accordingly (e.g., `GTE1-<uuid>`). If no prefix is set, the system generates a random one in the format `P-T-R-XXXX-`.

> **Base URL**: All operator endpoints are under `/api/v1`
>
> **Required Header**: `Authorization: Bearer <access_token>`

### 4.1 `POST /api/v1/operator/check-in`

Creates a new parking ticket for an incoming vehicle. Supports standard entry (by license plate) and registered customer entry (by customer code).

**Roles:** `GATE_STAFF`, `TENANT_OWNER`

**Request Body:**

| Field           | Type   | Required | Description                                                         |
|-----------------|--------|:--------:|---------------------------------------------------------------------|
| `vehicle_type`  | string | ✅        | Must be uppercase: `CAR`, `BIKE`, `TRUCK`, `BUS`, `VAN`            |
| `license_plate` | string | ⬜        | Vehicle plate number. Auto-uppercased. Can be omitted if `customer_code` is provided |
| `customer_code` | string | ⬜        | Code of a registered regular customer for discount application      |
| `notes`         | string | ⬜        | Optional internal note (max 255 chars)                              |

> If **neither** `license_plate` nor `customer_code` is provided, a guest placeholder plate `GUEST-XXXXXX` is auto-generated.

**Example Request:**
```json
{
  "vehicle_type": "CAR",
  "license_plate": "BA 1 CHA 4567"
}
```

**Example with Customer Code:**
```json
{
  "vehicle_type": "CAR",
  "customer_code": "REG-0042"
}
```

**Success Response `201`:**
```json
{
  "success": true,
  "ticket": {
    "ticket_id": "664b2f...",
    "ticket_number": "GTE1-xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "license_plate": "BA1CHA4567",
    "vehicle_type": "CAR",
    "check_in_time": "2026-05-29T04:30:00.000Z",
    "qr_code_url": "data:image/png;base64,...",
    "customer_name": null
  },
  "receipt": {
    "business_name": "City Parking Hub",
    "ticket_number": "GTE1-xxxxxxxx-...",
    "license_plate": "BA1CHA4567",
    "vehicle_type": "CAR",
    "entry_time": "2026-05-29 10:00:00",
    "qr_code_url": "data:image/png;base64,...",
    "printable_text": "========================================\n  CITY PARKING HUB\n========================================"
  }
}
```

**Common Errors:**

| Code | Reason |
|------|--------|
| `409` | Vehicle already has an active ticket |
| `404` | Customer code not found |
| `400` | Customer account is inactive/suspended |
| `403` | Operator is assigned to EXIT gate only |

### 4.2 `POST /api/v1/operator/scan`

Scans a ticket QR code or looks up by ticket number / license plate. Used at the exit gate to retrieve ticket details before processing check-out.

**Roles:** `GATE_STAFF`, `TENANT_OWNER`

**Request Body:**

| Field  | Type   | Required | Description                                              |
|--------|--------|:--------:|----------------------------------------------------------|
| `code` | string | ✅        | The scanned QR value, ticket number, or license plate    |

The backend resolves the identifier in this order:
1. Exact match on `ticket_number`
2. Case-insensitive match on `ticket_number` (lowercase)
3. Uppercase match on `license_plate`

**Example Request:**
```json
{
  "code": "GTE1-xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
}
```

**Success Response `200`:**
```json
{
  "success": true,
  "ticket": {
    "ticket_id": "664b2f...",
    "ticket_number": "GTE1-xxxx...",
    "license_plate": "BA1CHA4567",
    "vehicle_type": "CAR",
    "check_in_time": "2026-05-29T04:30:00.000Z",
    "check_out_time": null,
    "fare_amount": 0,
    "penalty_amount": 0,
    "discount_amount": 0,
    "status": "ACTIVE",
    "payment_method": null,
    "amount_received": null,
    "change_given": null,
    "transaction_reference": null,
    "notes": "",
    "customer_details": null,
    "qr_code_url": "data:image/png;base64,..."
  }
}
```

> **Tip:** Call `/scan` first to confirm ticket status, then call `/check-out` with the `ticket_id` to calculate the fare.

### 4.3 `POST /api/v1/operator/check-out`

Calculates the parking fare for an `ACTIVE` ticket, applies grace period and customer discounts, and transitions the ticket to `PENDING_PAYMENT`.

**Roles:** `GATE_STAFF`, `TENANT_OWNER`

**Request Body:**

| Field       | Type   | Required | Description                                                     |
|-------------|--------|:--------:|-----------------------------------------------------------------|
| `ticket_id` | string | ✅        | MongoDB ObjectId, ticket number (UUID), or license plate string |

**Example Request:**
```json
{
  "ticket_id": "664b2f..."
}
```

**Success Response `200`:**
```json
{
  "success": true,
  "summary": {
    "ticket_id": "664b2f...",
    "ticket_number": "GTE1-xxxx...",
    "license_plate": "BA1CHA4567",
    "vehicle_type": "CAR",
    "check_in_time": "2026-05-29T04:30:00.000Z",
    "check_out_time": "2026-05-29T07:30:00.000Z",
    "duration_minutes": 180,
    "chargeable_duration_minutes": 170,
    "rate_per_hour": 50,
    "subtotal": 150,
    "discount": 15,
    "total_amount": 135,
    "status": "PENDING_PAYMENT"
  }
}
```

> After a successful check-out, hand the `ticket_id` from this response directly to `/process-payment`.

**Common Errors:**

| Code | Reason |
|------|--------|
| `404` | No active ticket found for the provided identifier |
| `403` | Operator is assigned to ENTRY gate only |

### 4.4 `POST /api/v1/operator/process-payment`

Finalizes the transaction for a `PENDING_PAYMENT` ticket. Accepts cash or digital payment methods.

**Roles:** `GATE_STAFF`, `TENANT_OWNER`

**Request Body:**

| Field                   | Type   | Required       | Description                                              |
|-------------------------|--------|:--------------:|----------------------------------------------------------|
| `ticket_id`             | string | ✅              | MongoDB ObjectId of the `PENDING_PAYMENT` ticket         |
| `payment_method`        | string | ✅              | `CASH`, `ESEWA`, `KHALTI`, `FONEPAY`, `BANK_TRANSFER`    |
| `amount_received`       | number | ✅ (cash only) | Amount tendered by the customer (must be ≥ total due)    |
| `transaction_reference` | string | ⬜ (digital)   | Reference/UTR number from digital payment provider       |

**Example — Cash Payment:**
```json
{
  "ticket_id": "664b2f...",
  "payment_method": "CASH",
  "amount_received": 200
}
```

**Example — Digital Payment:**
```json
{
  "ticket_id": "664b2f...",
  "payment_method": "ESEWA",
  "transaction_reference": "TXN-20260529-ABC123"
}
```

**Success Response `200`:**
```json
{
  "success": true,
  "message": "Payment processed successfully.",
  "receipt": {
    "ticket_id": "664b2f...",
    "ticket_number": "GTE1-xxxx...",
    "total_due": 135,
    "payment_method": "CASH",
    "amount_received": 200,
    "change_given": 65,
    "printable_text": "┌─────────────────────┐\n│   PARKING RECEIPT   │\n│ ..."
  }
}
```

The `printable_text` field is formatted for **thermal printer output** and can be sent directly to a connected receipt printer.

**Common Errors:**

| Code | Reason |
|------|--------|
| `404` | Ticket not found or not in `PENDING_PAYMENT` status |
| `400` | Cash amount received is less than total due |
| `403` | Operator is assigned to ENTRY gate only |

### 4.5 `GET /api/v1/operator/stats`

Returns today's session summary for the current tenant. Operators can view their shift performance at a glance.

**Roles:** `GATE_STAFF`, `TENANT_OWNER`

**No request body required.**

**Success Response `200`:**
```json
{
  "success": true,
  "data": {
    "date": "2026-05-29",
    "_id": null,
    "totalVehicles": 45,
    "completedSessions": 38,
    "totalRevenue": 5400
  }
}
```

| Field               | Description                                            |
|---------------------|--------------------------------------------------------|
| `totalVehicles`     | Total check-ins today (all statuses)                   |
| `completedSessions` | Tickets that are NOT `ACTIVE` (checked out or paid)    |
| `totalRevenue`      | Sum of `fare_amount` for all tickets today             |

These routes under `/api/v1/parking` offer additional capabilities used primarily at exit gates.

### 5.1 `POST /api/v1/parking/lost-ticket`

Handles cases where a customer has lost their parking ticket. Creates a new ticket in `PENDING_PAYMENT` status with an assumed duration and a penalty charge.

**Roles:** `GATE_STAFF`, `TENANT_OWNER`

**Request Body:**

| Field                    | Type   | Required | Description                               |
|--------------------------|--------|:--------:|-------------------------------------------|
| `vehicle_type`           | string | ✅        | `CAR`, `BIKE`, `TRUCK`, etc.              |
| `license_plate`          | string | ✅        | Vehicle's visible license plate           |
| `assumed_duration_hours` | number | ✅        | Estimated hours parked (e.g., `3`)        |

**Example Request:**
```json
{
  "vehicle_type": "CAR",
  "license_plate": "BA 2 KHA 1234",
  "assumed_duration_hours": 4
}
```

**Success Response `201`:**
```json
{
  "success": true,
  "summary": {
    "ticket_id": "664c3a...",
    "ticket_number": "GTE1-xxxx...",
    "license_plate": "BA2KHA1234",
    "vehicle_type": "CAR",
    "assumed_duration_hours": 4,
    "base_amount": 200,
    "lost_ticket_penalty": 100,
    "total_amount": 300
  }
}
```

> After this, proceed to `/process-payment` with the returned `ticket_id`.

**Common Errors:**

| Code | Reason |
|------|--------|
| `409` | An active ticket already exists for this plate — cannot use lost ticket flow |
| `403` | Operator is assigned to ENTRY gate only |

### 5.2 `GET /api/v1/parking/tickets`

Returns a paginated list of all tickets for the current tenant. Useful for looking up historical or active records.

**Roles:** `GATE_STAFF`, `TENANT_OWNER`

**Query Parameters:**

| Param         | Type   | Default | Description                                                |

| `page`        | number | `1`     | Page number                                                |
| `limit`       | number | `20`    | Results per page (max: 100)                                |
| `status`      | string | *(all)* | Filter by `ACTIVE`, `PENDING_PAYMENT`, `PAID`              |
| `customer_id` | string | *(all)* | Filter by a specific customer's MongoDB ObjectId           |

**Example:**
```
GET /api/v1/parking/tickets?status=ACTIVE&page=1&limit=10
```

**Success Response `200`:**
```json
{
  "success": true,

    "page": 1,

    "totalPages": 9
  }

```

### 5.3 `GET /api/v1/parking/:id/receipt`

Fetches the formatted printable receipt for any ticket by its MongoDB ObjectId.

**Roles:** `GATE_STAFF`, `TENANT_OWNER`

**URL Parameter:** `:id` — The MongoDB ObjectId of the ticket.

**Example:**
```
GET /api/v1/parking/664b2f.../receipt
```

**Success Response `200`:**
```json
{
  "success": true,
  "printable_text": "========================================\n  CITY PARKING HUB\n..."
}
```

- If the ticket is `ACTIVE` or `PENDING_PAYMENT`: returns a **check-in style** receipt with entry time and QR prompt.
- If the ticket is `PAID`: returns a **full payment receipt** with duration, fare breakdown, and change given.

## 7. Offline Sync

The mobile app supports **offline-first** operation. When the device loses connectivity, all operations (check-ins, check-outs, payments) are cached locally and synced when the connection is restored.

### `POST /api/v1/sync/batch`

Submits a batch of offline operations to the server.

**Roles:** `GATE_STAFF`, `TENANT_OWNER`

**Request Body:**
```json
{
  "operations": [
    {
      "type": "CHECK_IN",
      "data": {
        "ticket_number": "GTE1-xxxx...",
        "license_plate": "BA1CHA4567",
        "vehicle_type": "CAR",
        "check_in_time": "2026-05-29T04:00:00.000Z",
        "notes": ""
      }
    },
    {
      "type": "CHECK_OUT",
      "data": {
        "ticket_number": "GTE1-xxxx...",
        "check_out_time": "2026-05-29T07:00:00.000Z",
        "fare_amount": 150,
        "discount_amount": 0
      }
    },
    {
      "type": "PAYMENT",
      "data": {
        "ticket_number": "GTE1-xxxx...",
        "payment_method": "CASH",
        "amount_received": 200,
        "change_given": 50
      }
    }
  ]
}
```

**Supported Operation Types:**

| Type         | Required Fields                                                             |
|--------------|-----------------------------------------------------------------------------|
| `CHECK_IN`   | `ticket_number`, `license_plate`, `vehicle_type`, `check_in_time`           |
| `CHECK_OUT`  | `ticket_number`, `check_out_time`, `fare_amount`, `discount_amount`          |
| `PAYMENT`    | `ticket_number`, `payment_method`, `amount_received`, `change_given`         |

**Success Response `200`:**
```json
{
  "success": true,
  "message": "Batch sync completed",
  "results": {
    "successful": 3,
    "failed": 0,
    "errors": []
  }
}
```

> **Idempotency:** `CHECK_IN` operations use upsert — sending the same ticket number twice will not create duplicates.

**Partial Failure Example:**
```json
{
  "results": {
    "successful": 2,
    "failed": 1,
    "errors": [
      {
        "ticket_number": "GTE1-yyyy...",
        "type": "CHECK_OUT",
        "error": "Ticket not found or already checked out"
      }
    ]
  }
}
```

## 8. Standard Fare Calculation Logic

Understanding how fare is calculated helps operators explain charges to customers.

```
1. Calculate total duration in minutes from check_in_time → check_out_time

2. If duration ≤ grace_period_minutes:
   → fare = 0 (free parking within grace period)

3. If duration > grace_period_minutes:
   → chargeable_minutes = total_minutes - grace_period_minutes
   → hours_to_charge = CEIL(chargeable_minutes / 60)  [always rounds UP]
   → base_fare = hours_to_charge × rate_per_hour

4. If customer has a discount:
   → discount_amount = base_fare × (discount_percentage / 100)
   → final_fare = base_fare - discount_amount

5. For LOST TICKET:
   → base_fare = assumed_duration_hours × rate_per_hour
   → total = base_fare + lost_ticket_penalty
```

**Example Calculation:**

| Parameter              | Value      |
|------------------------|------------|
| Vehicle type           | CAR        |
| Rate per hour          | Rs. 50     |
| Grace period           | 10 minutes |
| Duration parked        | 2h 35m (155 min) |
| Customer discount      | 10%        |

```
Chargeable minutes = 155 - 10 = 145 min
Hours to charge    = CEIL(145 / 60) = CEIL(2.41) = 3 hours
Base fare          = 3 × 50 = Rs. 150
Discount           = 150 × 10% = Rs. 15
Final charge       = Rs. 135
```

## 9. Ticket Lifecycle & Status Reference

```
  [VEHICLE ARRIVES]
        │
        ▼
   ┌─────────┐
   │  ACTIVE  │  ← Created at Check-In
   └────┬─────┘
        │  POST /check-out (fare calculated)
        ▼
┌─────────────────┐
│ PENDING_PAYMENT │  ← Awaiting payment at exit
└────────┬────────┘
         │  POST /process-payment
         ▼
      ┌──────┐
      │ PAID │  ← Transaction complete, vehicle exits
      └──────┘

[LOST TICKET FLOW]
  POST /lost-ticket → Creates ticket directly in PENDING_PAYMENT
        │
        ▼
  POST /process-payment → Moves to PAID
```

| Status            | Description                                       |
|-------------------|---------------------------------------------------|
| `ACTIVE`          | Vehicle is currently parked inside the facility   |
| `PENDING_PAYMENT` | Check-out calculated; awaiting payment            |
| `PAID`            | Payment received; transaction fully complete      |

## 10. Error Reference

All error responses follow this consistent structure:

```json
{
  "success": false,
  "message": "Human-readable error description"
}
```

| HTTP Status | Error Class       | Common Cause                                                      |
|-------------|-------------------|-------------------------------------------------------------------|
| `400`       | `ValidationError` | Invalid or missing required field (e.g., lowercase vehicle type, insufficient cash) |
| `401`       | `AuthError`       | Missing, expired, or invalid Bearer token                         |
| `403`       | `ForbiddenError`  | Correct token, but wrong role or wrong gate assignment            |
| `404`       | `NotFoundError`   | Ticket, customer, or rate record not found                        |
| `409`       | `ConflictError`   | Vehicle already has an active ticket                              |
| `500`       | Internal Error    | Unhandled server or database error; contact system administrator  |

## 11. Complete Operator Workflow Diagram

### Standard Entry → Exit → Payment

```
ENTRY GATE                           EXIT GATE
──────────────────                   ──────────────────────────────

1. Scan / enter plate
   POST /operator/check-in
   ─────────────────────              
   ← Returns: ticket_number           
              qr_code_url             
              printable receipt       

2. Print entry receipt
   (send printable_text to
    thermal printer)                  

3. Vehicle parks                     4. Vehicle arrives at exit
                                        POST /operator/scan
                                        ─────────────────────
                                        ← Returns: ticket details
                                                   current status

                                     5. Calculate fare
                                        POST /operator/check-out
                                        ─────────────────────────
                                        ← Returns: total_amount
                                                   discount
                                                   status: PENDING_PAYMENT

                                     6. Collect payment
                                        POST /operator/process-payment
                                        ──────────────────────────────
                                        ← Returns: change_given
                                                   printable receipt
                                                   status: PAID

                                     7. Print payment receipt
                                        Barrier opens → Vehicle exits
```

### Lost Ticket Flow

```
EXIT GATE
──────────────────────────────────────

1. Customer has no ticket
   Get vehicle type + license plate

2. Estimate duration from customer
   POST /parking/lost-ticket
   ────────────────────────────────
   ← Returns: total_amount (base + penalty)

3. Collect payment
   POST /operator/process-payment
   ────────────────────────────────
   ← Returns: PAID receipt

4. Barrier opens → Vehicle exits
```
